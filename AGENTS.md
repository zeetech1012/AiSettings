# Правила работы AI-агента (командный baseline)

> Правила для Claude Code. Подключаются через `@AGENTS.md`-импорт в `~/.claude/CLAUDE.md`
> либо копипастом из онбординг-гайда. Личные настройки и секреты сюда не попадают.

## Роль агента — заполняется в личном файле, не здесь

Роль определяет, **кем выступает агент**: уровень экспертизы и специализацию, стиль (сразу
к делу / развёрнуто), язык объяснений и кода, что флагировать до реализации (напр. flaky-тесты,
плохая поддерживаемость). Это персональная часть — в командный файл она не вписывается,
иначе копия разойдётся с baseline. Заполняется в `~/.claude/CLAUDE.md` над строкой `@AGENTS.md`
(образец — `claude/CLAUDE.example.md`).

> Всё, что ниже (правила написания кода, качество, дисциплина задач, Оккам, LLM-Wiki, handoff) —
> передаётся **как есть**.

## Дисциплина задач: баг = единица работы

- Нашёл дефект → **один** баг в трекере: root-cause + repro + предлагаемый подход + severity.
  Этого достаточно для разработчика. Тикеты-обёртки («реализовать фикс бага X») не заводить.
- Перед созданием — дедупликация: сверить с открытыми тикетами и wiki-списком багов
  (fulltext-поиск трекера ненадёжен); после transient-ошибки create проверить, не создался ли дубль.
- Прод-код QA не правит (баг → dev), repro остаётся красным до фикса.

## Роутинг по стекам

| Репозиторий | Стек | Заметки |
|---|---|---|
| `energy-app-native` | Kotlin (KMP + Compose + Kaspresso, L1-L2) + Swift/XCUITest (iOS, L3) + Allure | локаторы — единый `testTag`-контракт |
| `qa-monorepo` | Go: `resty` + `testify` + `ozontech/allure-go` + `coder/websocket` | backend black-box (API / контракты / E2E) |
| `auth_http` и др. Go-сервисы | Go: `net/http`, `httptest`, `testing`/testify | тесты сервиса изнутри |

Default для QA-задачи: бэкенд/контракт → Go (`qa-monorepo`); экран/Compose → Kotlin (KMP);
iOS-специфика (клавиатура, secure-field, permissions) → Swift/XCUITest. Неясно — спроси.
**Проектный instructions-файл репозитория главнее этого файла.**

## Как писать код

1. **Думай до кода.** Проговаривай допущения; несколько трактовок — покажи, не выбирай молча;
   есть путь проще — скажи; неясно — остановись и спроси.
2. **Простота.** Минимум кода под задачу: без спекулятивных фич, абстракций для одноразового
   кода и «конфигурируемости», которую не просили.
3. **Хирургические правки.** Трогай только нужное; не «улучшай» соседний код; стиль файла
   важнее твоих предпочтений; подчищай только мусор, созданный твоими же правками.
4. **Критерий успеха до кода.** «Почини баг» → «тест, воспроизводящий баг, затем зелёный»;
   формулируй проверяемые шаги и итерируй до verified.

## Правила качества (все стеки)

- Один класс/функция = одна ответственность.
- Никаких хардкодов URL / учётных данных / ID — только ENV, фикстуры, `BuildConfig`.
- Никаких sleep-ожиданий (`Thread.sleep` / `delay` / `time.Sleep`) — polling / framework-wait с таймаутом.
- Тесты независимы: без разделяемого мутабельного состояния и порядка выполнения.
- Flaky-тест = сломанный тест; не прятать за retry/xfail.
- Локаторы стабильные (`data-testid`, `testTag`), не XPath/CSS-класс/`R.id`.
- Каждый тест аннотирован для Allure (title/severity + epic/feature/story) и построен
  по Arrange-Act-Assert со steps; вложения (ответы API, скриншоты) — чтобы отчёт читался не-разработчиком.

## Чеклист ревью — флагируй автоматически

- sleep-ожидание → flaky-риск; хардкод URL/кредов → env-violation; нет Allure-аннотаций →
  reporting gap; XPath/CSS-локатор → fragile selector; фикстура с неверным scope → isolation risk;
  общее мутабельное состояние → order dependency; несколько логических assert без steps →
  debugging nightmare.
- При диагностике (flaky / красный тест / триаж) — принцип Оккама: простейшее объяснение,
  бьющееся с фактами, проверяй первым (skill `occams-razor`).

## Документация сервисов

- Источник истины по эндпоинтам/моделям — контекст-док сервиса (`docs/<service>.md`,
  генерируется skill'ом `service-doc` в репозитории сервиса).
- Дока отсутствует → **не угадывать** эндпоинты, поля и структуры ответов. Остановись
  и запроси контекст-док.

## LLM-Wiki — база знаний проекта

Знания о коде агент ведёт не в чатах, а в markdown-вики (паттерн Карпатого): код — неизменяемая
истина, вики — накапливаемый слой знаний по явной схеме.

- Два дерева по назначению: **рабочий слой** (планы, задачи, аудиты, отчёты — внутреннее) и
  **export-слой** (описание системы для коллег, выгружается в трекер/доки). Граница назначения =
  граница дерева; export не ссылается на рабочий слой.
- У каждого проекта — `index.md` (каталог заметок, one-line summary) и `log.md` (append-only
  хроника, ops: ingest / query / lint / scaffold).
- Операции: **ingest** (источник → страницы → index → log) · **query** (ответ из вики со ссылками) ·
  **lint** (битые ссылки, сироты, дрейф, противоречия).
- Ценность копится: каждая новая заметка перелинковывается с накопленным.

> Конкретные пути дерева и инструмент (Obsidian или иной) — в личной/командной настройке;
> здесь — принцип.

## Handoff — передача контекста между сессиями/людьми

- `./HANDOFF.md` в репозитории — **gitignored** ephemeral-кэш «возобновись здесь»
  (нет в `.gitignore` — добавить). Durable-копия — в личной базе знаний участника.
- В handoff: что сделано, что осталось, ключевые `file:line`, failed approaches, как проверить.
