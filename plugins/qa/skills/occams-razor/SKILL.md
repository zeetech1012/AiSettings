# Occam's Razor — эвристика диагностики

Когда несколько гипотез объясняют **одни и те же** факты одинаково хорошо —
проверяй первой ту, что требует **меньше новых допущений**. Это приоритизация
последовательности проверок, а **не** доказательство истины.

Применять ТОЛЬКО при равной объяснительной силе. Если факты реально требуют
сложного объяснения — бери сложное (см. анти-паттерн).

## Процесс

1. **Оставь подходящие.** Выпиши гипотезы, отбрось те, что не бьются с уликами
   (логи, тайминг, диффы, версии). Не подходит фактам → выбывает сразу.
2. **Посчитай допущения.** Для оставшихся — сколько новых, ничем не
   подтверждённых допущений нужно каждой.
3. **Проверь самую дешёвую первой.** Falsify её конкретным тестом/проверкой.
4. **Усложняй только после фальсификации.** Простая отпала → бери следующую по
   числу допущений. Не наоборот.

## QA-применение

- **Flaky-тест** → «гонка / `sleep` / shared state / неверный scope фикстуры в
  самом тесте» раньше «инфра / CI / сеть / сервис мигает». Тест виноват чаще.
- **Тест покраснел после изменения** → «контракт/ассерт/локатор сдвинулся»
  раньше «баг во фреймворке или рантайме».
- **Баг-кандидат на master** → простейшая причина, что бьётся с фактами
  (напр. «стенд на дефолтном секрете», «подпись не проверяется», «фильтр по не
  тому полю»), раньше многоходовых цепочек.
- **Триаж** (`triage`/`diagnose`/`debug-test`) — ранжируй гипотезы по числу
  допущений, начинай отладку с верхней.
- **Расследование 500/паники** → «необработанный edge во входе» (пустой
  параметр, короткий заголовок, not-found) раньше «повреждение БД / OOM».

## Анти-паттерны

- Прятать за «простотой» улики, которые **требуют** сложности (оверсимплификация).
- Применять при НЕравной объяснительной силе → выбрать «просто, но неверно».
- Путать «меньше допущений» с «привычнее / проще понять».
- Цепляться за простую гипотезу, когда она уже **фальсифицирована** тестом.
- Считать Бритву доказательством, а не порядком проверки.
