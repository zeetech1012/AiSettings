# Huly Team Work Reference

## Личная настройка

1. Собери `huly-mcp` и добавь блок `huly` из `mcp/mcp.json`, `mcp/codex.toml` или
   `mcp/gemini.settings.json` в конфиг своего клиента.
2. Укажи только личные `HULY_URL`, `HULY_TOKEN` и `HULY_WORKSPACE`. Токен не добавляй в Git,
   чат или настройки плагина.
3. Перезапусти клиента и запроси список доступных проектов через MCP. Если проект не виден,
   запроси доступ у администратора Huly. Не пытайся обходить RBAC.
4. Для личной работы вызови `huly-team-work`, например: «создай задачу в проекте API».

## Выбор типа

Три task types (модель с 2026-07-20). ЗНИ — не тип, а входной статус Эпика «Запрос на изменение».

| Сигнал в запросе | Тип | Стартовый статус |
|---|---|---|
| Заявка на изменение / инициатива, которую надо согласовать | Эпик | `Запрос на изменение` |
| Согласованная крупная цель, требующая декомпозиции | Эпик | `БЭКЛОГ` |
| Есть конкретная работа с ясным результатом | Задача | `БЭКЛОГ` |
| Фактическое поведение не совпадает с ожидаемым | Баг | `БЭКЛОГ` |

Заявка на изменение и работа по ней — один тикет (Эпик): согласование продвигает его из
«Запрос на изменение» в «БЭКЛОГ», отклонение — в «ОТМЕНЕНО». Отдельную сущность-заявку не заводим.

Не превращай Баг в Задачу и не создавай отдельную задачу только для исправления уже описанного
дефекта. Свяжи Баг с затронутой работой или Эпиком, если родитель известен.

## Шаблоны тела

### Эпик — заявка на изменение (стартовый статус «Запрос на изменение»)

```text
Обоснование (зачем):
Требования:
Влияние на пользователей и системы:
Критерии приёмки:
Инициатор:
```

### Эпик — согласованная инициатива (стартовый статус «БЭКЛОГ»)

```text
Цель / ценность:
Границы:
Результат:
План декомпозиции:
Критерии готовности:
```

### Задача

```text
Ценность:
Результат:
Критерии приёмки:
Зависимости и ограничения:
```

### Баг

```text
Environment:
Steps to reproduce:
Actual result:
Expected result:
Severity:
Attachments:
```

## Перед созданием

1. Назови проект и проверь, что он доступен твоей учётной записи.
2. Найди похожие открытые тикеты. При совпадении обнови или свяжи существующий, не создавай
   дубль.
3. Проверь обязательные поля по шаблону.
4. Покажи preview с типом, проектом, статусом и родителем.
5. Создавай тикет только после явного подтверждения автора запроса.

## Движение по статусам

Наборы сняты с живого трекера; ПРОЕКТ - SOFT и ПРОЕКТ - HW настроены по отдельности (расхождение
осознанное). Свежие статусы huly-mcp может не резолвить по имени — тогда передавай `status` по id
из `get_project_type`.

| Тип | Допустимый путь (SOFT) | Отличие HW |
|---|---|---|
| Эпик | `Запрос на изменение → БЭКЛОГ → ПЛАН → В РАЗРАБОТКЕ → В ТЕСТИРОВАНИИ → Протестировано (Verified) → СДЕЛАНО / ОТМЕНЕНО` | активные стадии `РЕАЛИЗАЦИЯ → ТЕСТИРОВАНИЕ`, без Verified |
| Задача | `БЭКЛОГ → ПЛАН → В РАЗРАБОТКЕ → В ТЕСТИРОВАНИИ → Протестировано (Verified) → СДЕЛАНО / ОТМЕНЕНО` | активные стадии `РЕАЛИЗАЦИЯ → ТЕСТИРОВАНИЕ` |
| Баг | `БЭКЛОГ → ПЛАНИРОВАНИЕ → В ИСПРАВЛЕНИИ → ИСПРАВЛЕН / ОТМЕНЕНО` | без отличий |

Не меняй assignee, проект или родительскую связь без явной просьбы. Перед закрытием проверь
критерии приёмки для Задачи или Expected result для Бага.

## Границы ответственности

- `huly-team-work` ведёт личные тикеты через личный токен.
- `huly-service-issue` работает только от имени intake-бота (сервисный токен).
- `huly-doc-sync` синхронизирует документацию и не создаёт рабочие тикеты.
