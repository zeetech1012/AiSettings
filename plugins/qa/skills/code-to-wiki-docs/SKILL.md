# Code → Wiki Articles

Превращает исходный код репозитория в набор **вики-статей** (Markdown), пригодных для базы
знаний (Docmost/Confluence/GitHub wiki). Не путать с [service-doc] (тот делает узкий
QA-контекст для написания тестов) — здесь цель шире: читаемая документация для людей.

## Два пути генерации

### A. Автоматический — Skill Seekers (только для подходящих источников)
Установлен инструмент `skill-seekers` (v3.x). Скрейпит источник и пакует в скилл/доки:
```bash
skill-seekers create ./<repo> --name <slug>     # → структурированное знание в output/<slug>
skill-seekers create https://docs.<...>          # docs-сайт по URL
skill-seekers package output/<slug> --target claude   # → Claude skill (ZIP); таргеты: gemini|openai|langchain|cursor
skill-seekers quality output/<slug>/SKILL.md     # оценка качества
```
**Где работает хорошо:** docs-сайты (scrape по URL), Python/JS/TS-репо (их AST он парсит),
PDF/видео/Swagger-страницы.

> ⚠️ **Не понимает Kotlin/Swift.** Проверено на `energy-app-native` (KMP): из 855 файлов
> распарсил 0, граф зависимостей пуст, «design patterns» — мусор по именам. В LOCAL-режиме
> (без API-ключа) он просто запускает агента, который **пересказывает уже существующие
> `output/docs/`** в один SKILL.md — нового знания о коде не даёт, плюс рискует галлюцинировать
> ссылки. Для Kotlin/KMP-исходников → идти путём B. Авто-путь применять к docs-сайтам и
> Python/JS/TS (напр. `qa-monorepo`). Любой результат вычитывать вручную (шаг 4).

### B. Ручной — точечные статьи (когда нужен контроль и точность)

**Шаг 1 — Разведка кода.** Читать в порядке: entry point (`main`/`App`/`MainActivity`) →
навигация/роутинг → слои (ui/domain/data) → DI-модуль → ключевые модели → конфиги. Фиксировать
факты с путями `file:line` — их потом проверять.

**Шаг 2 — Разбить по аудиториям (spaces).** Не валить всё в один файл. Типовая раскладка:
| Статья | Аудитория | Содержание |
|---|---|---|
| `product_overview.md` | бизнес/менеджеры | что умеет, какие сценарии, платформы |
| `app_flow.md` | все | карта экранов и навигации (Mermaid + скриншоты) |
| `architecture_context.md` | разработка/QA | слои, DI, транспорты, ключевые файлы (факты «проверено») |
| `<feature>_data_flow.md` | разработка | data-flow конкретной фичи (Mermaid) |
| `build_and_run.md` | разработка/DevOps | предусловия, команды, типичные ошибки |
| `user_guide.md` | конечные юзеры | пошагово, простым языком |
Добавить `README.md`-индекс, разводящий статьи по пространствам.

**Шаг 3 — Писать в Docmost-формате.** Применять правила из [docmost-qa-strategy] §1:
`<details>` для длинных блоков, callouts `> [!info]/[!warning]`, мета-блок `> **Аудитория:**`,
таблицы, Mermaid для схем. Заголовки `##/###` без пропусков уровней — для оглавления.

**Шаг 4 — Верификация (обязательно).** Каждый технический факт сверять с исходником перед
записью; помечать «проверено». **Не выдумывать** эндпоинты, поля, пути. Если факт не
подтверждён кодом — писать прямо «требует уточнения», а не догадку.

**Шаг 5 — Перекрёстные ссылки.** В конце каждой статьи — секция «Связанные документы» со
ссылками на смежные. Имена файлов — нижний регистр + underscore (`app_flow.md`,
`build_and_run.md`).

## Куда класть результат
- В этом проекте канонический каталог вики — `output/docs/` (см. его `README.md`-индекс).
- Эфемерные/служебные доки (session-логи) — не в вики; держать вне коммита.

## Anti-patterns
- Один гигантский `README` вместо разбивки по аудиториям.
- Факты «по памяти» без сверки с кодом → устаревшая/ложная дока.
- Markdown, не рендерящийся в Docmost (вложенный код без пустой строки после `<summary>`).
- Дублирование `service-doc` (QA-контекст) — у него другая цель и формат.
